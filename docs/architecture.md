# 架构说明

`api-blackbox-tester` 是一个由多个独立 Skill 组成的 Codex 插件。各 Skill 按测试生命周期拆分，使测试方案、测试用例评审、执行验证和最终报告可以独立审查、重复执行，并保持清晰的证据链。

## 组件关系

```text
api-blackbox-tester
        |
        | 根据用户意图选择阶段
        v
+-----------------------------+
| api-blackbox-test-planner   |
+-----------------------------+
        |
        | 测试方案、用例 ID、数据计划、编排链路
        v
+-----------------------------+
| api-blackbox-test-reviewer  |
+-----------------------------+
        |
        | 评审结论、退回项、执行关注项
        v
+-----------------------------+
| api-blackbox-test-executor  |
+-----------------------------+
        |
        | curl/grpcurl 证据、DB 校验、失败原因
        v
+-----------------------------+
| api-blackbox-test-reporter  |
+-----------------------------+
        |
        | 结论非通过时输出根因、解决建议、复验范围
        v
+-----------------------------+
| 开发修复 / 解除阻塞          |
+-----------------------------+
        |
        | 修复后回到 executor 复验，再进入 reporter
        +---------------------> api-blackbox-test-executor
```

## Skill 职责

### `api-blackbox-tester`

统一编排入口。它判断用户需要测试方案、测试用例评审、执行验证、测试报告，还是完整的阶段化流程。

该 Skill 不承载具体测试细节，只负责阶段选择、流程串联和交付物衔接要求。

### `api-blackbox-test-planner`

测试方案阶段。它在可用时读取需求、项目上下文、路由定义、DTO、中间件、已有测试、API 文档、配置和数据库访问逻辑。

它输出以下内容：

- 测试范围和假设。
- 兼容性验证。
- 测试数据计划。
- 功能测试用例。
- 边界测试。
- 安全测试。
- 集成或接口编排链路。
- 补充验证项。
- 性能测试建议。

### `api-blackbox-test-reviewer`

测试用例评审阶段。它在 planner 和 executor 之间提供执行前门禁，由资深测试工程师、资深产品经理、资深软件工程师、安全工程师四个角色分别评审测试方案，再通过小组讨论形成一致结论。

它输出以下内容：

- 风险分级覆盖矩阵、需求追踪、可执行性 preflight、安全建模和 Anti-Happy-Path 检测。
- 四个角色的独立评审意见。
- 小组讨论博弈结论。
- 最终评审结果：通过、有条件通过、不通过或阻塞。
- 必须退回 planner 完善项。
- executor 执行关注项。

只有评审结果为通过或有条件通过时，executor 才能执行。评审不通过时必须回到 planner 完善测试方案后重新评审。覆盖配额和安全建模按接口风险适用，低风险接口可以给出不适用说明，高风险接口不能用泛化豁免跳过关键异常、安全或数据一致性评审。

### `api-blackbox-test-executor`

执行验证阶段。它以真实接口行为为准，并在需要数据库准备或校验时优先使用 `db-mcp`。执行前必须确认 reviewer 阶段已经通过或有条件通过。

它记录以下证据：

- 执行环境。
- 数据准备。
- 请求命令。
- 响应结果。
- 数据库验证。
- 编排链路步骤。
- 清理状态。

### `api-blackbox-test-reporter`

报告阶段。它汇总 planner 输出和 executor 证据，评估：

- 需求覆盖率。
- 用例执行覆盖率。
- 接口覆盖率。
- 边界测试覆盖率。
- 安全测试覆盖率。
- 集成链路覆盖率。
- 数据场景覆盖率。
- 兼容性覆盖率。
- 性能测试建议覆盖率。

缺少真实执行证据时，reporter 不得将变更标记为通过。

当结论为失败、部分通过或阻塞时，reporter 必须输出可执行的下一动作输入，包括问题根因分析、解决建议、责任阶段和复验范围。失败进入开发修复，部分通过补齐未执行或未覆盖项，阻塞必须先自动尝试解除；只有自动解除失败且缺口必须由用户或外部系统提供时，才进入 `WAIT_USER_INPUT` 暂停态。后续必须回到 executor 执行 HTTP/gRPC 复验，再由 reporter 基于复验证据重新评估，直到结论为通过。

## 阶段交付契约

阶段之间依赖稳定 ID 和明确证据：

| 生产方 | 输出 | 消费方 |
| --- | --- | --- |
| Planner | `tests/【需求】_YYYYMMDD/测试方案.md`，包含用例 ID、链路 ID、数据计划、预期响应和 DB 状态 | Reviewer |
| Reviewer | `tests/【需求】_YYYYMMDD/测试用例评审.md`，包含四角色意见、讨论结论、最终评审结果和执行关注项 | Executor |
| Executor | `tests/【需求】_YYYYMMDD/执行记录.md`，包含 curl/grpcurl 命令、响应、DB 校验、失败原因 | Reporter |
| Reporter | `tests/【需求】_YYYYMMDD/测试报告.md`，包含覆盖率、测试结论、根因分析、解决建议、剩余风险、发布建议和复验计划 | 用户、发布流程或开发修复阶段 |

四个阶段必须复用同一需求目录。`【需求】` 取用户需求的简短标题，`YYYYMMDD` 取执行当天日期或用户指定日期。

## 外部依赖

`db-mcp` 对规划阶段不是必需项，但在执行阶段需要数据库准备或校验时应使用。它可用于查看表结构、创建或复用测试数据、验证副作用，以及清理测试归属的数据。

HTTP 接口执行通常使用 `curl`。gRPC 接口执行通常使用 `grpcurl`。

## 设计约束

- 每个 Skill 聚焦单一阶段。
- 下游阶段可消费的输出格式必须保持清晰。
- 四个阶段都必须写入对应 Markdown 文件，不能只在对话中输出。
- executor 必须受 reviewer 评审结果门禁约束。
- 结论以证据为准，不以推测为准。
- 报告结论不是通过时，必须回到开发修复、补齐覆盖或解除阻塞，再进行 executor 复验和 reporter 再评估。
- 所有未通过、部分通过、阻塞和剩余风险必须包含根因分析、解决建议和复验范围。
- 环境、凭证或依赖缺失时，应判定为阻塞，而不是通过；但阻塞不是完成态，必须先自动补救，无法自行解除时进入 `WAIT_USER_INPUT`。
- 未明确授权时，不执行破坏性操作。

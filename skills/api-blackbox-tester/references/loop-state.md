# 闭环状态机和 artifact 判定

## 闭环优先级

每次响应前必须读取当前测试目录中最新 artifact，先判定闭环状态，再选择下一阶段。当前测试目录指最新或用户指定的 `tests/【需求】_YYYYMMDD/`，artifact 包括 `测试方案.md`、`测试用例评审.md`、`执行记录.md`、`测试报告.md` 和复验记录。

如果当前会话或当前测试目录已经存在同一需求的 `测试报告.md`，且报告结论为 `失败`、`部分通过` 或 `阻塞`，则进入闭环优先模式：

- 用户说“下一步怎么做”、“继续”、“处理”、“修复”、“复验”等，必须解释为继续闭环执行，而不是只输出 prompt 或建议。
- 不得停留在“建议开发修复”；必须自动进入 `开发修复 -> 最小验证/单测 -> 自动编译/重启最多 3 次 -> executor 复验 -> reporter 再评估`。
- 只有用户明确说“只给建议”、“不要修改代码”、“暂停”、“先不要修”时，才允许停止在说明阶段。
- 如果修复后需要编译或重启服务，应先完成代码修复和最小验证/单测，再根据项目脚本、README、进程管理配置或既有上下文自动尝试重新编译并重启服务。最多重试 3 次，每次必须记录命令、结果和失败原因；只有 3 次都失败或缺少必要权限/外部控制面时，才允许标记状态为 `FIXED_WAIT_RESTART` 和 `待用户重启服务`。用户确认重启后，必须自动继续 executor 复验。
- 如果报告中的失败、部分通过或阻塞问题不属于开发修复范围，应自动推进测试数据准备、环境/运维、需求澄清、测试方案补充或安全授权等责任阶段；无法自行解除时，必须明确标记阻塞条件和解除后的复验范围。

## 闭环状态机

- `PLANNED`：已有 `测试方案.md`，下一步 reviewer。
- `REVIEWED_PASS`：`测试用例评审.md` 结论为通过或有条件通过，下一步 executor。
- `EXECUTED`：已有 `执行记录.md`，下一步 reporter。
- `REPORT_FAILED`：`测试报告.md` 结论为失败、部分通过或阻塞，下一步开发修复或解除阻塞。
- `FIXED_WAIT_RESTART`：代码已修复且最小验证/单测已完成，agent 已自动尝试编译/重启服务最多 3 次但未成功，或缺少必要权限/外部控制面，等待用户确认重启。
- `RESTARTED`：用户确认服务已重启，下一步 executor 复验。
- `REPORT_PASS`：`测试报告.md` 结论为通过，闭环结束。

## 最新 artifact 判定算法

每次响应前按以下顺序判定当前测试目录和闭环状态：

1. 用户明确指定 `tests/【需求】_YYYYMMDD/` 时，优先使用该目录。
2. 用户未指定目录时，优先使用当前会话最近一次写入或提到的测试目录。
3. 如果会话中没有明确目录，扫描当前项目根目录下 `tests/*/`，按目录内阶段文件的最新修改时间选择最近的测试目录；阶段文件包括 `测试报告.md`、`执行记录.md`、`测试用例评审.md`、`测试方案.md` 和复验记录。
4. 在同一目录内，按文件修改时间读取最新 artifact；如果时间接近或冲突，按状态优先级判定：`REPORT_FAILED`/`REPORT_PASS` 高于 `EXECUTED`，`EXECUTED` 高于 `REVIEWED_PASS`，`REVIEWED_PASS` 高于 `PLANNED`。
5. 读取 `测试报告.md` 时必须解析最终结论字段；结论为 `失败`、`部分通过` 或 `阻塞` 时进入 `REPORT_FAILED`，结论为 `通过` 时进入 `REPORT_PASS`。
6. 如果 `测试报告.md` 缺失但有 `执行记录.md`，状态为 `EXECUTED`；如果只有评审通过或有条件通过的 `测试用例评审.md`，状态为 `REVIEWED_PASS`；如果只有 `测试方案.md`，状态为 `PLANNED`。
7. 如果 artifact 内容与时间顺序冲突，例如较新的执行记录对应较旧报告，必须说明冲突，并优先继续到 reporter 重新评估，不得沿用旧通过结论。

状态判定以最新 artifact 为准；如果 artifact 之间冲突，必须说明冲突证据，优先使用时间最新且证据最完整的文件，并在需要时标记为阻塞。

# 测试用例评审模板和评分模型

## 量化判定规则

必须给出 `0-100` 的质量评分，并列出扣分原因。评分用于辅助一致决策，不能替代硬性拒绝规则和风险判断。

建议评分权重：

- 影响点与影响范围覆盖：12 分。
- 覆盖矩阵：10 分。
- 接口 x 字段语义等价类矩阵：15 分。
- 跨接口一致性：7 分。
- Traceability：10 分。
- 可执行性 Preflight：10 分。
- Anti-Happy-Path：10 分。
- STRIDE 安全建模：10 分。
- 对抗性辩论有效性：5 分。
- 数据计划、清理和断言质量：9 分。
- 小组讨论结论可执行性：2 分。

## Must Pass

以下条件必须满足，否则不得给出 `通过`。不适用项必须说明原因；对高风险接口不接受泛化豁免。

- 每个核心接口必须有正常路径和异常路径用例。
- 资深软件工程师必须完成影响点与影响范围评估，且所有关键影响点必须映射到测试用例、矩阵单元或编排链路。
- 每个核心接口必须有接口 x 字段语义等价类矩阵，且每个必测矩阵单元必须关联用例和断言。
- 逻辑类似接口必须有跨接口一致性用例，且每个参与接口都必须独立执行验证；不得用一个接口的测试代替另一个接口。
- 每个核心需求必须至少关联 1 个用例。
- 每个用例必须有明确响应断言；有副作用时必须有 DB 断言。
- 写操作必须评审幂等性、重复提交或重放行为。
- 受保护接口必须包含鉴权和权限测试。
- 用户可控输入必须包含至少 1 个安全测试。
- 测试数据必须说明构造或获取方式，以及清理策略。
- executor 必需的 base URL、凭证、数据和依赖缺失时，必须标记阻塞或退回 planner，不得直接通过。

## Reject If

触发以下任一规则时，最终结果必须为 `不通过` 或 `阻塞`：

- 无测试用例、无用例 ID 或用例无法追踪到需求/接口。
- 用例只有场景描述，没有明确断言。
- 写接口没有数据准备、DB 断言或清理策略。
- 安全相关接口没有安全用例。
- 核心或高风险接口异常用例占比明显不足，且没有明确豁免理由。
- 核心或高风险接口检测到 Happy Path 偏置且未补充反向路径。
- 缺少接口 x 字段语义等价类矩阵，或核心接口必测矩阵单元无关联用例。
- 对逻辑类似接口未设计跨接口一致性用例，且没有明确业务差异证据。
- 缺少执行所需关键环境、凭证或依赖信息，且无法在 executor 前补齐。
- 缺少影响点与影响范围评估，或关键影响点没有测试覆盖且没有明确豁免依据。
- 核心或高风险接口的对抗性辩论发现有效遗漏，但未补充用例、断言或退回 planner。

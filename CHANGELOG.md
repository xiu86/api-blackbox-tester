# 变更日志

本文件记录项目的重要变更。

本项目遵循语义化版本。

## [0.7.0] - 2026-06-17

### 新增

- 新增结构化路由影响分析契约，planner 必须优先选择当前项目可用的结构化路由分析工具，不限定具体工具；没有可用工具时退回代码 diff、路由定义、API 文档、代码搜索、项目约定和 Agent 推理。
- 新增 `contracts/`，集中定义 artifact、评审角色、证据包、门禁和术语契约。
- 新增 `scripts/validate_all.py` 分层门禁，覆盖结构、结构化路由影响分析、内容、多角色评审、追踪、证据、通过门禁和非通过闭环。
- 新增结构化路由影响分析正反向夹具，校验结构化分析输出发现的接口必须进入变更代码影响接口清单。

### 变更

- planner、reviewer、executor、reporter 和统一入口同步结构化路由影响分析、变更代码影响接口清单、风险驱动用例、断言计划、证据采集计划、证据包和三段式可信度门禁。
- reviewer 从四角色评审升级为五角色多角色评审，新增测试证据审计员。
- reporter 可信度章节统一为覆盖可信度、证据可信度和综合可信度。
- 兼容旧门禁脚本入口，内部转发到新的分层门禁。

## [0.6.1] - 2026-06-09

### 变更

- reporter 报告和终端摘要增加测试成果汇报，必须输出已执行测试用例、发现 Bug、已修复 Bug、复验通过 Bug、待修复 Bug、阻塞 Bug、疑似 Bug 和非 Bug 风险数量。
- reporter 明确 Bug 数按问题或根因去重，不按失败用例数简单相加。
- README、使用指南、架构说明和插件元数据同步测试成果汇报能力。

## [0.6.0] - 2026-06-01

### 新增

- planner 增加业务场景拆解，要求根据需求、技术方案、验收标准、状态机、角色权限、数据生命周期和接口链路拆分所有业务场景。
- planner 增加场景 x 边界/异常覆盖矩阵，核心业务场景不能只测 Happy Path，必须覆盖适用边界值、异常路径、权限/状态/数据变体和跨接口失败链路。
- reviewer 增加业务场景拆解评审和场景 x 边界/异常覆盖矩阵评审，核心场景缺边界或反向路径时不得通过。
- executor 增加场景 x 边界/异常覆盖执行状态回填。
- reporter 增加业务场景覆盖、场景 x 边界/异常覆盖和测试结果可信度评分。
- 新增 artifact 契约和测试结果可信度评分规则。
- 新增 `validate_test_artifacts.py` 和 `check_report_gate.py` 自动门禁脚本，报告结论为通过前必须校验阶段文件和通过条件。

### 变更

- tester 统一入口将业务场景拆解、场景边界覆盖和自动门禁脚本纳入全流程硬追踪对象。
- 更新各阶段 `agents/openai.yaml` 默认提示词，确保本地 Codex 调用时触发新的业务场景和可信度门禁。

## [0.5.0] - 2026-05-13

### 新增

- reporter 报告增加测试执行汇总，必须统计计划、已执行、通过、未通过、跳过、阻塞和未执行用例数量。
- reporter 报告增加未通过/阻塞根因简述，要求把失败根因和下一步动作汇总到报告前部。
- report 模板增加终端输出摘要，要求报告写入后在终端同步输出报告路径、用例汇总、未通过根因、覆盖率摘要、最终结论、下一步动作和剩余风险。

### 变更

- reporter Skill 和默认提示词明确要求报告内容与终端摘要数字、结论和根因保持一致。

## [0.4.2] - 2026-05-13

### 变更

- tester 增加非通过结论强制闭环门禁，明确只有 `REPORT_PASS` 是完成态。
- loop-state 增加 `WAIT_USER_INPUT` 暂停态，阻塞报告必须先尝试自动解除，不能直接收尾。
- executor 增加阻塞前自动补救规则，要求在标记阻塞前搜索凭证、定位 DB、检查服务、构造测试数据或处理依赖。
- reporter 增加阻塞解除计划，缺 token、DB 不匹配、服务未部署、数据缺失和依赖不可用都必须先自动尝试处理。
- execution/report 模板增加自动补救、阻塞解除计划和 `WAIT_USER_INPUT` 记录字段。

## [0.4.1] - 2026-05-12

### 变更

- reviewer 明确 `Happy Path 偏置` 定义，区分“必须覆盖正常路径”和“只验证正常路径”的质量风险。
- Anti-Happy-Path Gate 调整为风险类别覆盖优先，异常用例占比仅作为辅助信号，不能替代关键风险类别判断。
- reviewer 模板增加核心或高风险接口关键风险类别缺失的 Reject If 规则。
- Happy Path 偏置检测模板增加正向路径覆盖、异常占比辅助判断和缺失关键风险类别字段。

## [0.4.0] - 2026-05-12

### 新增

- planner 增加影响范围初判，要求记录直接影响、潜在扩散影响、证据来源和高/中/低置信度。
- reviewer 增加独立评审模板 `references/review-template.md`，集中维护评分模型、Must Pass、Reject If 和输出结构。
- executor 增加执行记录模板 `references/execution-template.md`，集中维护请求记录、矩阵回填、跨接口一致性、编排链路、DB 验证和清理状态格式。
- tester 增加共享闭环状态文件 `references/loop-state.md`，统一闭环优先级、状态机和最新 artifact 判定算法。

### 变更

- 非通过报告闭环调整为自动进入 `开发修复 -> 最小验证/单测 -> 自动编译/重启最多 3 次 -> executor 复验 -> reporter 再评估`，仅在自动编译/重启失败或缺少权限时进入 `FIXED_WAIT_RESTART`。
- executor 发现测试范围缺口时，不再直接执行原方案；关键范围缺口回 planner 完整补方案，非关键补证项回 reviewer 评审。
- reporter 和 tester 共用闭环状态定义，避免 artifact 判定和状态机逻辑分叉。
- reviewer 主 Skill 精简为流程和规则摘要，详细评分与模板下沉到 reference，降低维护成本。

## [0.3.0] - 2026-04-29

### 新增

- 新增 `api-blackbox-test-reviewer` Skill，在 planner 和 executor 之间增加测试用例评审门禁。
- 评审由资深测试工程师、资深产品经理、资深软件工程师、安全工程师四个角色分别提出意见，再通过小组讨论形成最终结论。
- 新增 `tests/【需求】_YYYYMMDD/测试用例评审.md` 阶段交付物。
- reviewer 增加风险分级覆盖矩阵、需求追踪、可执行性 preflight、安全建模、Anti-Happy-Path 检测和量化判定依据。
- reporter 增加非通过结论的根因分析、解决建议、责任阶段和复验计划要求。

### 变更

- 完整流程调整为 `planner -> reviewer -> executor -> reporter`。
- executor 必须在评审结果为通过或有条件通过后才执行；不通过或阻塞时退回 planner 完善。
- reviewer 的异常路径、安全评审和评分规则按接口风险等级适用，避免机械套用固定配额。
- 测试报告结论不是通过时，完整流程必须回到开发修复或解除阻塞，并在修复后执行接口复验和报告再评估。
- README、使用指南、架构说明、发布检查和插件提示词同步四阶段流程。

## [0.2.3] - 2026-04-28

### 变更

- 明确三个测试阶段必须写入对应 Markdown 文件。
- 默认输出目录调整为 `tests/【需求】_YYYYMMDD/`。
- 约定 planner 写入 `测试方案.md`，executor 写入 `执行记录.md`，reporter 写入 `测试报告.md`。
- 更新 README、使用指南、架构说明、发布流程和插件默认提示词。

## [0.2.2] - 2026-04-27

### 新增

- 新增项目文档：README、使用指南、架构说明、发布流程、贡献指南、行为准则、安全策略和许可证。

### 既有能力

- 通过 `api-blackbox-tester` 编排接口黑盒测试流程。
- 通过 `api-blackbox-test-planner` 输出编码前测试方案。
- 通过 `api-blackbox-test-executor` 执行真实请求和数据库验证。
- 通过 `api-blackbox-test-reporter` 输出基于证据的覆盖率报告。
